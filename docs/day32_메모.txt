────────────────────────────────
1️⃣ nn.Module 기본 구조 — 이거 외워둬
────────────────────────────────
class 모델이름(nn.Module):
    def __init__(self):
        super().__init__()        # ← 반드시 필요!
        self.encoder = nn.Linear(입력, 출력)
        self.decoder = nn.Linear(출력, 입력)

    def forward(self, x):
        x = torch.relu(self.encoder(x))
        x = self.decoder(x)
        return x

핵심:
  super().__init__() 없으면 → 가중치 등록 안 됨 → 에러
  forward() 는 model(x) 호출 시 자동으로 실행됨

────────────────────────────────
2️⃣ AutoEncoder 구조
────────────────────────────────
쉽게 말하면:
  압축했다가 → 다시 복원하는 모델

  입력(8) → 인코더 → 압축(4) → 디코더 → 복원(8)

왜 이상탐지에 쓰냐?
  정상 로그로만 학습 → 정상은 잘 복원
  이상 로그는 본 적 없어 → 복원 못함 → 오류 큼
  오류가 크면 = 이상!

────────────────────────────────
3️⃣ 학습 루프 5단계 — 순서 절대 외워
────────────────────────────────
for epoch in range(epochs):
    optimizer.zero_grad()      # 1) 기울기 초기화
    output = model(data)       # 2) 순전파
    loss = criterion(output, data)  # 3) loss 계산
    loss.backward()            # 4) 역전파
    optimizer.step()           # 5) 가중치 업데이트

왜 zero_grad()가 필요해?
  PyTorch는 기울기를 기본적으로 누적해
  초기화 안 하면 이전 기울기가 쌓여서
  잘못된 방향으로 가중치가 업데이트됨

────────────────────────────────
4️⃣ model.train() vs model.eval()
────────────────────────────────
model.train()  → 학습 모드
  Dropout 활성화 (랜덤하게 뉴런 끔)
  BatchNorm 학습 통계 사용

model.eval()   → 추론 모드
  Dropout 비활성화 (모든 뉴런 켬)
  BatchNorm 저장된 통계 사용

→ 추론할 때 eval() 안 쓰면 결과가 매번 달라짐!

────────────────────────────────
5️⃣ torch.no_grad() 왜 쓰나?
────────────────────────────────
추론(eval) 할 때는 학습이 필요 없어
기울기 계산을 꺼서 메모리/속도 절약

with torch.no_grad():
    output = model(data)  # 기울기 계산 안 함

## 투 포인터 알고리즘
================================================================

────────────────────────────────
핵심 구조 — 외워둬
────────────────────────────────
def two_pointer(scores, target):
    left = 0
    right = len(scores) - 1

    while left < right:           # for문 아니고 while!
        current = scores[left] + scores[right]
        if current == target:
            return left, right    # 정답!
        elif current > target:
            right -= 1            # 합이 크면 오른쪽 줄이기
        else:
            left += 1             # 합이 작으면 왼쪽 키우기

    return None

────────────────────────────────
왜 정렬된 배열에서만 가능해?
────────────────────────────────
정렬된 배열에서는:
  합 > target → right-=1 → 반드시 합이 작아짐
  합 < target → left+=1  → 반드시 합이 커짐

방향이 예측 가능하기 때문에 O(n)으로 탐색 가능!

정렬 안 됐으면:
  right를 줄여도 합이 커질 수 있어 → 방향 예측 불가
  → 투 포인터 사용 불가

시간복잡도:
  이중 for문: O(n²)
  투 포인터 : O(n)  ← 정렬된 배열 조건 필요

## 🇯🇵 면접 Q&A — 소리내서 읽어봐
================================================================

Q: optimizer.zero_grad()をなぜ毎回呼ぶ？
A: PyTorchは勾配をデフォルトで累積するため、
   呼ばないと前の勾配が足し合わされ、
   正しい方向に重みを更新できなくなります。

Q: model.eval()とmodel.train()の違いは？
A: eval()はDropoutを無効化し、
   推論結果を安定させます。
   train()はDropoutを有効化して学習を行います。

Q: AutoEncoderで異常検知できる理由は？
A: 正常データのみで学習したモデルは
   正常データを再構成できますが、
   異常データは再構成できません。
   この再構成誤差の差を異常スコアとして使います。

Q: なぜソート済み配列でTwo Pointersが使えるの？
A: ソート済みなら合計がtargetより大きい時に
   rightを減らすと必ず合計が小さくなります。
   この予測可能な方向性があるためO(n)で探索できます。
