────────────────────────────────
1️⃣ F1 기준선 테스트 — 실무 필수 패턴
────────────────────────────────
쉽게 말하면:
  "코드가 바뀌어도 성능이 일정 수준 이상인지
   자동으로 확인하는 감시원"

왜 필요해?
  코드 수정 → 성능이 떨어져도 모름
  F1 기준선 테스트가 있으면 → 자동으로 알림

이걸 "성능 회귀 테스트(Performance Regression Test)" 라고 해
일본 면접에서 쓰면 눈빛이 달라지는 키워드!

코드:
  assert best_metrics['f1'] >= 0.8

기준선 설정 원칙:
  너무 높게 (≥0.99) → seed 달라지면 불안정
  너무 낮게 (≥0.1)  → 아무 모델이나 통과 → 의미없음
  적당하게 (≥0.8)   → 실무 수준 보장 ✅

────────────────────────────────
2️⃣ scores.shape 비교 — 헷갈리기 쉬운 것
────────────────────────────────
shape는 튜플이야:

  scores = torch.mean(..., dim=1)  # shape: (200,)

  assert scores.shape == 200       # ❌ 숫자와 비교 → False
  assert scores.shape == (200,)    # ✅ 튜플과 비교 → True
  assert scores.shape == (len(normal),)  # ✅ 더 유연한 방법

────────────────────────────────
3️⃣ 최소 구간 커버 알고리즘 — 그리디
────────────────────────────────
쉽게 말하면:
  점들을 정렬하고 첫 점에서 구간 시작
  구간 밖의 점이 나오면 → 그 점에서 새 구간 시작

핵심 코드:
  scores = sorted(scores)
  count = 1                      # 첫 구간 포함!
  cover_until = scores[0] + k

  for num in scores[1:]:
      if num > cover_until:
          count += 1
          cover_until = num + k  # 새 점 기준으로!

자주 틀리는 것:
  count = 0 으로 시작 → 첫 구간 안 셈 → 항상 1 작게 나옴
  cover_until += k   → 이전 끝에서 더함 → 잘못된 구간
  cover_until = num + k → ✅ 새 점 기준이 맞아!

시간복잡도:
  sorted(): O(n log n)
  for문   : O(n)
  전체    : O(n log n)

## 🇯🇵 면접 Q&A
================================================================

Q: テストにF1の基準値を設ける理由は？
A: コードが変わったとき、性能が低下しても
   気づけないことがあります。
   F1≥0.8の基準線を設けることで、
   性能回帰(Performance Regression)を
   自動的に検知できます。

Q: scores.shapeの比較でなぜタプルを使うのですか？
A: PyTorchのshapeはタプル型で返されるため、
   数値と比較するとFalseになります。
   (200,)のようにタプルで比較する必要があります。

Q: グリーディアルゴリズムで区間カバーを解く考え方は？
A: ソート後、最初の点から区間を開始し、
   区間外の点が現れたら新しい区間を開始します。
   各ステップで最善の選択をするグリーディ法です。
   時間計算量はO(n log n)です。
